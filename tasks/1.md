
# 需求(有的实现, 这个是大的方向, 不一定都实现)
+ 让他不断干活, 领取我的任务
	+ 相当于有一个任务池这样的东西, 是以工作区为
+ 可以同时控制多个开发环境
	+ 例如本机环境/ 远程 ssh的某个目录 / 远程ssh 的某个容器的某个目录
+  Git worktree 实现并行化
	+ 面临一个大的任务,把它拆分为多个小任务, 使用git worktree 并行处理
	+ 针对的是一个工作区中的项目 (例如一个容器里面某个目录)
+ tmux 管理多终端(这个有必要吗, 还是说通过过程日志/结果日志来管理)
	+ 例如开启的不同会话, 都维持以一个 tmux session 来统一管理,方便断线重连
+ 过程总结提炼
	+ 在不断开发的时候, 进行升级迭代
	+ 并且防止它变得过度臃肿
+ 每个项目有一个 ./task 目录, 下面保存着任务对应的计划md文件
+ 有一个GPU资源查询的过程
	+ 例如 /data/projectA 这个项目,  和 /data/projectB 都是位于同一个linux server 中, 但是他们是不同的workspace, 我的项目都默认使用 GPU, nvdia 显卡, 如果两个项目同时在执行。修改代码同时可以。但是如果在运行测试的时候，两个项目同时进行。他们就会发生资源的争抢。所以说这部分也需要。合理的处理。就是说当他们使用相同的GPU, 可能是需要进行一些资源探测
